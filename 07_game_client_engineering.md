## 设计模式

### q001

**问题：** 单例模式如何实现？

[返回问题总览](07_game_client_engineering.md#问题总览)

**回答：**

- C++ 可用函数局部 static 实现懒加载单例，C++11 起初始化线程安全。
- C# 可用 static readonly 或 Lazy<T>。
- 构造函数私有化，禁止外部 new。
- 需要考虑销毁顺序、跨场景生命周期和测试替换。
- 实现 单例模式 前要先定义输入、输出、核心状态和生命周期。
- 优先选能保证复杂度和可维护性的数据结构，而不是先写流程代码。
- 主流程要拆成初始化、更新/查询、异常处理和释放四部分。
- 边界条件要提前列出，例如空数据、重复调用、越界、取消、对象已销毁。
- 如果运行在游戏帧循环里，要说明更新频率、是否分帧、是否影响主线程。
- 设计模式服务于解耦和复用，不应为了模式而模式。
- 可通过依赖关系、测试替换、生命周期和模块边界验证设计质量。
- 收束时给出一个项目中的使用或排查场景，可以把答案从概念背诵提升到工程经验。

**面试补充：**

- 追问方向：复杂度、边界条件、并发/生命周期问题和如何测试。
- 实现题要说清楚核心数据结构，否则容易被认为只会写流程。
- 落地检查：可通过依赖关系、测试替换、生命周期和模块边界验证设计质量。

**示例：**

```cpp
class Manager {
public:
    static Manager& Instance() {
        static Manager instance;
        return instance;
    }
private:
    Manager() = default;
};
```

### q045

- 把必须资源和非必须资源分层。
- 优先下载登录/首场景必要内容，其他资源后台补齐。
- 使用断点续传、压缩和 CDN 加速。
- 展示准确进度，并避免解压/校验阻塞主线程。
- 优化/排查题第一步永远是定位瓶颈，先拿数据再改代码。
- 观察指标要围绕版本管理、资源依赖、代码更新、校验、回滚和兼容性，不要只看单一现象。
- 常见流程是：复现问题、采集数据、缩小范围、提出假设、做 A/B 验证、回归测试。
- 只会下载文件不够，必须有 manifest、hash、依赖、灰度、回滚和兼容策略。
- 修复优先级按收益/风险排序，先做低风险高收益项，再做结构性改造。
- 热更新影响线上修复速度、活动交付、用户等待时间和平台合规。
- 可用断点续传测试、损坏包校验、版本回滚、弱网下载和线上灰度验证。
- 热更新的定义：在不发整包的情况下更新资源、配置或脚本逻辑。

**面试补充：**

- 追问方向：你如何证明这个优化有效，以及是否引入画质/逻辑回归。
- 不要把经验当结论，优化题要始终回到数据和验证。
- 落地检查：可用断点续传测试、损坏包校验、版本回滚、弱网下载和线上灰度验证。
